# Retail Data Warehouse — ETL Pipeline with Star Schema

An end-to-end ETL pipeline that ingests raw retail transaction data, transforms it into a dimensional star schema, loads it into a DuckDB analytical warehouse, and runs automated data quality checks.

## Architecture

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Raw CSV     │────▶│   Extract    │────▶│  Transform   │────▶│    Load      │
│  Sources     │     │  & Validate  │     │  Star Schema │     │   DuckDB     │
└─────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
                                                                      │
                                                               ┌──────▼──────┐
                                                               │   Data      │
                                                               │   Quality   │
                                                               │   Checks   │
                                                               └─────────────┘
```

## Dataset

Uses the **Superstore Sales** dataset — a widely-used retail dataset containing ~10,000 orders across furniture, office supplies, and technology categories with customer, product, shipping, and financial attributes.

A synthetic version is auto-generated by the pipeline for portability (no external download required).

## Pipeline Stages

### 1. Extract
- Reads raw CSV source files from `data/raw/`
- Validates schema (column names, data types, nullability)
- Logs row counts and anomalies
- Rejects malformed records to a quarantine file

### 2. Transform
- Generates surrogate keys for all dimensions
- Builds `dim_customer`, `dim_product`, `dim_date`, `dim_ship_mode`
- Constructs `fact_sales` with foreign key references
- Derives calculated fields: `discount_flag`, `ship_category`, `fiscal_quarter`
- Handles SCD Type 1 (overwrite) for dimension attributes

### 3. Load
- Creates/replaces tables in SQLite warehouse
- Enforces primary key and foreign key constraints
- Loads dimension tables first, then fact table
- Logs load statistics and timing

### 4. Data Quality
- **Completeness**: No NULL values in required columns
- **Uniqueness**: Primary keys are unique across all tables
- **Referential Integrity**: All fact table FKs exist in dimension tables
- **Range Checks**: Quantities > 0, discounts between 0–1, dates in valid range
- **Aggregation Tests**: Fact table totals reconcile with source data

## Sample Analytical Queries

```sql
-- Top 10 customers by lifetime value
SELECT c.customer_name, c.segment,
       SUM(f.sales_amount) AS lifetime_value,
       COUNT(DISTINCT f.order_id) AS total_orders
FROM fact_sales f
JOIN dim_customer c ON f.customer_key = c.customer_key
GROUP BY c.customer_name, c.segment
ORDER BY lifetime_value DESC
LIMIT 10;

-- Monthly revenue trend with YoY comparison
SELECT d.year, d.month_name, d.month,
       SUM(f.sales_amount) AS revenue,
       SUM(f.profit) AS profit,
       ROUND(SUM(f.profit) / NULLIF(SUM(f.sales_amount), 0) * 100, 1) AS margin_pct
FROM fact_sales f
JOIN dim_date d ON f.date_key = d.date_key
GROUP BY d.year, d.month_name, d.month
ORDER BY d.year, d.month;

-- Product category performance with discount impact
SELECT p.category, p.sub_category,
       SUM(f.sales_amount) AS revenue,
       SUM(f.profit) AS profit,
       AVG(f.discount) AS avg_discount,
       SUM(CASE WHEN f.discount_flag = 1 THEN f.sales_amount ELSE 0 END) AS discounted_revenue
FROM fact_sales f
JOIN dim_product p ON f.product_key = p.product_key
GROUP BY p.category, p.sub_category
ORDER BY revenue DESC;
```

## Key Design Decisions

1. **Star schema over flat table** — Demonstrates dimensional modeling knowledge; enables efficient analytical queries with clear grain definition
2. **SQLite for portability** — Zero-dependency embedded DB; swap to DuckDB or Redshift for production analytical workloads with minimal code changes (DDL + connection only)
3. **Surrogate keys** — Insulates fact table from source system ID changes; standard warehousing practice
4. **Synthetic data generator** — Makes the project fully self-contained; no external downloads needed
5. **Quarantine pattern** — Bad records are isolated, not silently dropped; mirrors production data engineering practices

## CI/CD — GitHub Actions

The pipeline runs automatically via GitHub Actions:

| Trigger | When |
|---------|------|
| **Scheduled** | Every Monday at 6 AM UTC |
| **Push** | On changes to `src/`, `config.yaml`, or `requirements.txt` on `main` |
| **Pull Request** | On PRs targeting `main` |
| **Manual** | Via "Run workflow" button in GitHub Actions UI |

The workflow runs two parallel jobs:

- **Run ETL Pipeline** — Executes the full pipeline and uploads the warehouse + processed data as downloadable artifacts (retained 30 days). Quarantine reports are uploaded separately for audit visibility.
- **Run Tests** — Executes the pytest suite to validate all stages independently.
